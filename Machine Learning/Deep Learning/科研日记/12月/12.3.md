### 🛑 第一关：OpenMMLab 的版本命名“陷阱”

*   **现象：**
    在使用 `mim download` 下载模型配置时，报错 `ValueError: Expected configs...`，提示找不到 `votenet_8x8...`。
*   **原因：**
    **代码库更新导致的不兼容。** OpenMMLab 在新版本（v1.1+）中修改了配置文件的命名规范，将 `8x8`（8卡x8样本）改为了 `8xb8`。网上的旧教程没有及时更新。
*   **教训：**
    当开源库报错“找不到配置”时，不要死磕教程里的名字，去报错日志给出的**“白名单（Expected keys）”**里找最像的那个名字。
*   **最终解：**
    将命令修改为 `votenet_8xb8_scannet-3d.py`。

---

### 🛑 第二关：无头服务器与 Open3D 的“水土不服”

*   **现象：**
    运行 `demo/pcd_demo.py` 时，报错 `NotImplementedError` 或 `dash` 相关错误。
*   **原因：**
    **依赖库过于臃肿。** 最新版的 Open3D 为了支持 Jupyter Notebook 的网页可视化，强制引入了 `dash` 和 `plotly`。而你的服务器是 **Headless（无图形界面）** 的，无法启动这些 web 组件，导致程序崩溃。
*   **教训：**
    在服务器上跑深度学习，环境**“越老越稳”**。不要追求最新版库。
*   **最终解：**
    暴力降级。卸载 `open3d` 和 `dash`，安装纯净版的 `open3d==0.13.1`。

---

### 🛑 第三关：本地可视化的“格式巴别塔”

*   **现象：**
    1.  本地 Python 脚本弹窗报错 `GLFW Error: Wayland`（Linux 显示协议冲突）。
    2.  生成的 `.ply/.obj` 文件在 CloudCompare 里打不开，提示 `incompatible file types`。
*   **原因：**
    **数据结构不通用。**
    *   **Wayland 错误：** 是因为 Open3D 的老旧 OpenGL 接口不支持新的 Linux 桌面协议。
    *   **文件打不开：** 是因为 Open3D 导出的“LineSet（线框）”是一种特殊结构，而 CloudCompare（以及大多数 Mesh 查看软件）默认只认“点（Point）”和“面（Mesh）”，不认单纯的“线”。
*   **教训：**
    **“所见即所得”的数据最安全。** 当高级格式（LineSet）不兼容时，就退回到最底层、最通用的格式（XYZ 纯文本点云）。
*   **最终解：**
    放弃“画线”，改用“画密集的点”。将 3D 框离散化为一串红色的点，保存为 `.xyz` 格式。这打破了所有软件的兼容性壁垒。
